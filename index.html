The 0/1 Knapsack Problem

A classic combinatorial optimization problem

What is it?

You have a bag (knapsack) with a maximum weight capacity W. You are given n items, each with a weight and a value. Your goal is to select a subset of items that maximizes total value without exceeding the weight limit. Each item can either be taken or left behind (0 or 1) -- you cannot take a fraction of an item.

Formal Definition

maximize ∑ vi · xi

subject to ∑ wi · xi ≤ W

where xi ∈ {0, 1}

Why does it matter?

The knapsack problem appears everywhere: budget allocation, cargo loading, resource scheduling, portfolio optimization, and even cutting stock. It is NP-hard in general, meaning there is no known algorithm that solves all instances quickly. However, when the capacity W is an integer, Dynamic Programming solves it in O(nW) time -- pseudo-polynomial, but fast enough in practice.

Why not just be greedy?

A greedy approach (always take the item with the best value-per-weight ratio) seems intuitive but fails. Consider: capacity = 10, items A($6, 1kg), B($10, 2kg), C($12, 3kg), D($15, 5kg). Greedy ranks by ratio (A=6, B=5, C=4, D=3) and takes A+B+C = $28 at 6kg. But the optimal is B+C+D = $37 at 10kg. Greedy left $9 on the table because it favored lightweight items and missed the heavier combination that fills the bag perfectly.

The DP Solution

Dynamic Programming builds a table M[i][w] where each cell represents the best value achievable using the first i items with capacity w. For each item, we compare two choices:

M[i][w] = max(

M[i-1][w],// skip item i

vi + M[i-1][w - wi]// take item i

)

The final answer is M[n][W]. To find which items were selected, we backtrack through the table: if M[i][w] differs from M[i-1][w], item i was taken.

O(n · W)
